# Knowledge Summary
Anthropic’s Claude skill guide is most valuable not because it teaches one Claude-specific file format, but because it offers a durable design principle for reusable AI workflow systems: keep the always-loaded layer small, load deeper instructions only when relevant, and separate workflow logic from supporting references. The Claude implementation uses a folder with `SKILL.md` plus optional `scripts/`, `references/`, and `assets/`, and relies on progressive disclosure through YAML frontmatter, the main skill body, and linked files.

The broader lesson is transferable across LLMs. A robust cross-LLM setup should not be built as one long master prompt. It works better as a layered operating system: a short global core, a small number of workflow playbooks, supporting reference files read only when needed, and lightweight task-specific session briefs. This reduces token waste, improves consistency, and lowers maintenance drift across platforms.

This note translates the Claude skill concepts into a reusable personal architecture for ChatGPT, Claude, Gemini, or similar systems, while also capturing key failure modes, privacy boundaries, and a reusable file structure.

# Extracted Details

## Anthropic guide: the transferable fundamentals
- A Claude skill is a folder containing:
  - `SKILL.md` as the required main file
  - optional `scripts/`
  - optional `references/`
  - optional `assets/`
- The guide’s core design principles are:
  - **Progressive disclosure**
  - **Composability**
  - **Portability**
- Progressive disclosure works across three levels:
  1. YAML frontmatter, always loaded and used for routing
  2. `SKILL.md` body, loaded when the skill is relevant
  3. linked files, opened only as needed
- The practical goal is to minimise token usage while preserving specialised expertise.
- The guide recommends starting with **2 to 3 concrete use cases**, not with a general ambition to “make the model smarter”.
- YAML frontmatter is treated as critical because it determines whether the skill is invoked.
- The `description` field must include both:
  - what the skill does
  - when to use it
- The guide also emphasises a clean file structure, strict naming rules, and avoiding unnecessary files such as an internal `README.md`.

## Durable cross-LLM pattern
The Claude file format should not be copied blindly into every model. The more durable pattern is architectural:

1. **Router layer**
   - a short description of what the module does
   - when to use it
   - what kinds of user requests should trigger it

2. **Playbook layer**
   - the actual workflow
   - ordered steps
   - review checks
   - output standards
   - common failure prevention

3. **Reference layer**
   - examples
   - templates
   - style guides
   - definitions
   - caveats
   - metric interpretation notes

4. **Tool-policy layer**
   - what tools are allowed
   - what tools are preferred
   - what tools are prohibited or unnecessary

5. **Session-brief layer**
   - objective
   - audience
   - deliverable
   - constraints
   - what changed for this task

This pattern preserves the main benefit of Claude skills, namely progressive disclosure, without depending on one vendor’s exact implementation.

## Main failure modes
Common failure modes when building skill-like systems for LLMs:

- **One giant omnibus prompt**
  - mixes routing, execution, reference material, examples, and maintenance logic
  - harder to maintain
  - more likely to be inconsistently followed

- **Too many overlapping skills**
  - vague boundaries between modules
  - duplicated instructions
  - unclear activation criteria
  - lower reliability

- **Storing too much static knowledge in live instructions**
  - long context loads
  - unnecessary token use
  - instructions become harder to update

- **Organising by topic instead of workflow**
  - creates fragmented modules
  - weak routing
  - too much overlap between files

- **Blending stable and volatile context**
  - long-term rules and one-off task details get mixed together
  - makes the system harder to reuse cleanly

- **Treating examples as instructions**
  - can create overfitting to sample wording
  - weakens generality

## Best approach: layered operating system
A practical personal setup works best as a layered system.

### Layer 1: Global core
Short, stable defaults that apply nearly everywhere:
- language preferences
- tone
- evidence and uncertainty rules
- anti-fabrication rules
- high-level output expectations

### Layer 2: Workflow playbooks
A small number of repeatable work modes, for example:
- research synthesis
- marketing reporting
- strategic communications
- review and quality control

### Layer 3: Reference files
Detailed material not needed every time:
- output formats
- confidence labels
- masked examples
- metric definitions
- style examples

### Layer 4: Session briefs
Per-task context:
- objective
- audience
- output needed
- constraints
- caveats
- urgency

### Layer 5: Maintenance layer
Used outside normal runtime:
- evaluation set
- change log
- version notes
- failure tracking
