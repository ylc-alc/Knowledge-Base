# Knowledge Summary
The key skill in working well with AI is not instantly knowing the perfect question. It is learning how to turn a vague discomfort, poor result, or messy situation into a clear problem statement. The same applies when working with humans. Better questions usually come from structured diagnosis rather than sudden insight.

# Why This Matters
Many people are told to "ask better questions", but that advice is incomplete. In practice, the real difficulty is that the user often does not yet know the true cause of the problem. This leads to shallow prompts, unclear requests, and wasted effort. A better approach is to diagnose the situation first, then ask for help in a more precise way.

# Extracted Details
- Most weak questions fail because the speaker does not yet know the real issue.
- Symptoms are often mistaken for causes.
- Several problems are often bundled together and treated as one.
- Success criteria are often undefined.
- In AI work, apparent prompt problems are frequently source, workflow, reasoning, or evaluation problems.
- Stronger requests usually separate:
  - current state
  - desired state
  - suspected cause
  - evidence
  - requested help

# Core Principle
Do not focus first on:
- "What is the perfect prompt?"

Focus instead on:
- "What exactly is going wrong?"
- "What am I trying to achieve?"
- "What kind of help do I need right now?"

# Key Framework

## 1. Start from the trigger
Begin with the observed friction or failure.

Ask:
- What happened that felt wrong?
- What result did I get?
- Why is that result unsatisfactory?

Example:
- Weak: "How do I write a better prompt?"
- Better: "My AI summary is shallow and misses the main argument."

## 2. Describe the gap
Make the gap explicit:
- What did I get?
- What did I want instead?
- What is missing between the two?

This converts vague dissatisfaction into something actionable.

## 3. Identify the level of the problem
Classify the issue before trying to fix it.

Possible levels:
- Input problem: wrong or incomplete source material
- Framing problem: unclear task definition
- Reasoning problem: weak logic or analysis
- Output problem: wrong structure, tone, or depth
- Workflow problem: the process itself is poorly designed

Many people try to solve all of these as if they were only prompt wording issues.

## 4. Define success before asking for help
A strong request should state:
- what good looks like
- what should be included
- what should be excluded
- how the result will be judged

Without this, even a competent answer may still feel wrong.

## 5. Ask for diagnosis before solution
When the issue is still unclear, start with diagnosis.

Example:
- Less effective: "Fix this."
- More effective: "Diagnose why this is weak first. List the top likely causes. Then suggest fixes."

This is often the highest-value shift when using AI.

# Problem Statement Template
Use this whenever your request feels unclear:

I am trying to [goal]  
For [audience or use case]  
But I am getting [current result]  
I think this may be because [suspected cause]  
I need help with [specific task]

Example:
I am trying to create a useful knowledge note from an article for future study.  
But I am getting a shallow summary based only on screenshot content.  
I think this may be because the workflow is not reading the full article.  
I need help redesigning the extraction logic and prompt.

# Five-Question Routine
Before asking AI or another person for help, answer these first:

1. What am I trying to do?
2. What is happening instead?
3. What do I think is causing that?
4. What evidence do I have?
5. What exactly do I want help with?

This is a simple repeatable way to move from confusion to a usable brief.

# Common Failure Modes
- Treating the symptom as the real problem
- Asking for too much in one step
- Failing to specify audience or purpose
- Not defining success criteria
- Mixing extraction, analysis, and writing into a single request
- Assuming the issue is prompt wording when it is actually a source or workflow issue
- Asking for a solution before understanding the problem type

# AI-Specific Application
When working with AI, useful meta-questions include:
- What is the actual task here?
- What information is missing?
- What assumptions am I making?
- What type of problem is this?
- Should this be solved in one pass or multiple passes?
- What would an expert check before trusting this output?
- Is this mainly a sourcing, reasoning, framing, or formatting issue?

A practical sequence is:
1. diagnose
2. solve
3. refine

# Human Collaboration Application
The same logic works with colleagues, clients, managers, and collaborators.

Useful clarifying questions:
- What decision are we trying to make?
- What would change if we knew the answer?
- What is currently failing?
- What evidence do we already have?
- What is still unknown?
- Which part is actually blocking progress?

This is why strong managers and analysts often appear good at "asking the right questions". In practice, they are often running a diagnosis process.

# Example Use Cases

## Writing
Problem:
- "This draft feels weak."

Better framing:
- "This draft explains the facts, but the argument is unclear for a senior stakeholder audience. Please diagnose whether the issue is structure, evidence, or positioning."

## Research
Problem:
- "Help me with this topic."

Better framing:
- "I need to understand the main mechanisms, the strongest evidence, and the unresolved questions on this topic. Separate established findings from speculation."

## Analysis
Problem:
- "Why is this dashboard bad?"

Better framing:
- "This dashboard shows metrics, but it does not help me decide what action to take. Diagnose whether the issue is metric choice, missing segmentation, weak comparison, or lack of context."

## Debugging
Problem:
- "My code is not working."

Better framing:
- "This script runs but produces the wrong output after the merge step. Help me isolate whether the issue is input data, join logic, or downstream transformation."

## AI Workflow Design
Problem:
- "The note generated from the article is too shallow."

Better framing:
- "The output seems to summarise only visible screenshot content rather than the full page. Diagnose whether the issue is source acquisition, prompt design, or output structure."

# Reusable Prompt Patterns

## Pattern 1: Diagnose first
"Before proposing a solution, diagnose the top three likely causes of the problem. Then recommend the most likely fix."

## Pattern 2: Clarify the gap
"Here is what I have, here is what I wanted, and here is the gap. Help me determine whether the problem is source quality, task framing, reasoning, or output structure."

## Pattern 3: Separate stages
"First extract the facts. Then analyse them. Then write the final output for the target audience."

## Pattern 4: Define the standard
"Use these criteria for judging quality: [criteria]. Explain where the current result falls short."

# Key Takeaways
- Good questions are usually built through diagnosis, not discovered instantly.
- The real skill is problem definition, not clever wording.
- Most weak requests come from unclear goals, bundled issues, or undefined success.
- Better requests distinguish current state, desired state, cause, evidence, and need.
- In AI work, source quality and workflow design often matter more than prompt phrasing alone.
- Asking for diagnosis before solution is one of the most effective habits to build.

# Actionable Tips
- Name the exact failure mode before revising a request.
- Rewrite vague complaints into diagnostic questions.
- Define what a good answer should contain before asking for one.
- Separate extraction, analysis, and writing when needed.
- Keep a habit of rewriting poor requests into better ones after failed attempts.
- Use the five-question routine whenever you feel stuck.

# Mini Checklist
Before sending a request, check:

- Do I know what I am trying to achieve?
- Have I described what is going wrong?
- Have I identified what type of problem this is?
- Have I defined what a good answer looks like?
- Should I ask for diagnosis first before asking for a solution?

# One-Sentence Reminder
The right question usually does not appear at the start. It is developed by clarifying the goal, diagnosing the gap, and naming the kind of problem you are actually dealing with.
